# FC Bayern Munich - Stats Portal & API 🔴⚪

Production-ready веб-приложение для управления статистикой, матчами и игроками футбольного клуба "Бавария" Мюнхен. 
*Этот файл также служит личным конспектом и справочником по архитектуре современных Python-приложений.*

---

## 📚 Теоретическая база (Словарь разработчика)

### 1. Архитектура и Разделение слоев (Layered Architecture)
Код не пишется в одном файле `main.py`. Мы разделяем логику, чтобы проект можно было легко масштабировать:
* **`src/api/` (Роутеры):** "Официанты". Принимают запросы из браузера, передают их дальше, забирают готовый ответ и отдают пользователю. Ничего не знают про базу данных.
* **`src/db/` (База Данных и Модели):** "Склад". Здесь описано, как таблицы физически выглядят на жестком диске (ORM Модели).
* **`src/schemas/` (Pydantic Схемы):** "Таможня". Проверяют все данные, которые приходят из интернета и уходят в интернет. Защищают сервер от плохих данных.
* **`src/core/` (Ядро):** Настройки, безопасность, чтение паролей из `.env`.

### 2. Базы Данных: ORM и SQLAlchemy
* **Проблема:** Python не понимает язык SQL, на котором говорят базы данных (PostgreSQL). Писать SQL-запросы вручную — долго и небезопасно.
* **Решение:** Мы используем **ORM (Object-Relational Mapping)** — библиотеку `SQLAlchemy`. 
* **Как работает:** Мы пишем обычный класс на Python (например, `class Player`), а SQLAlchemy сама, "под капотом", переводит это в SQL-код и создает таблицу. 

### 3. Базы Данных: Миграции (Alembic)
* **Что это:** Это "Git для базы данных". 
* **Зачем нужно:** Если мы хотим добавить игроку новое поле (например, `is_injured`), мы не можем просто удалить старую таблицу и создать новую — мы потеряем всех текущих игроков. 
* **Как работает:** Alembic сравнивает наши классы Python (`models.py`) с тем, что сейчас есть в PostgreSQL. Находит отличия и пишет скрипт (миграцию), который аккуратно добавляет новую колонку, не удаляя старые данные.

### 4. Pydantic Схемы vs SQLAlchemy Модели
Важно не путать:
* **SQLAlchemy Модель (`models.py`):** Описывает, как данные лежат **внутри базы данных** (включает системные поля, связи, кэши).
* **Pydantic Схема (`schemas.py`):** Описывает, как данные летят **через интернет** в формате JSON. Например, в Схеме мы можем скрыть пароль пользователя, чтобы он не улетел в браузер, хотя в Модели он есть.

### 5. Инфраструктура: Docker
* **Что это:** Изолированный "террариум" для программ.
* **Зачем нужен:** Чтобы не устанавливать тяжелую базу данных PostgreSQL прямо на свой Windows/Linux со всеми ее зависимостями. Мы просто запускаем контейнер, который живет в своем пузыре и имеет всё необходимое для работы. Гарантирует, что на сервере база запустится точно так же, как и на домашнем ПК.

### 6. Frontend: Jinja2 и Tailwind CSS
* **Jinja2:** Шаблонизатор. Позволяет писать логику (циклы `for`, условия `if`) прямо внутри HTML-кода. FastAPI берет HTML, выполняет логику Jinja (подставляет данные из базы) и отдает браузеру готовую страницу.
* **Tailwind CSS:** Utility-first CSS фреймворк. Вместо того чтобы писать отдельные файлы со стилями, мы пишем классы прямо в HTML (например, `class="bg-red-600 text-white"`). Это сильно ускоряет верстку.

## 💻 Шпаргалка: Команды для работы

**1. Инфраструктура (Docker):**
```bash
docker compose up -d    # Запустить базу данных в фоне
docker compose stop     # Остановить базу (данные СОХРАНЯЮТСЯ)
docker compose down -v  # ПОЛНОСТЬЮ УДАЛИТЬ базу и все данные (Опасно!)
```
**2. Запуск локального сервера (FastAPI):**
```bash
source .venv/bin/activate         # Активация окружения (всегда сначала!)
uvicorn src.main:app --reload     # Запуск сервера
```
**3. Миграции (Alembic):**
***Выполнять только после внесения изменений в файл models.py!***
```bash
alembic revision --autogenerate -m "Что изменили"  # Создать скрипт миграции
alembic upgrade head                               # Применить скрипт в базу
```
